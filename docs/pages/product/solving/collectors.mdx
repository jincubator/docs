# Data Collection

The implemented collector boundary ingests external market-state changes, normalizes them into protocol-independent components, updates graph views, and identifies affected candidate routes for evaluation.

The design separates ingestion, state processing, graph maintenance, route cataloguing, evaluation, and execution evidence. This makes it possible to test each boundary without presenting a route evaluation as an opportunity or an execution.

Live data-provider configuration, filtering thresholds, cache policy, queue priorities, and operating values are not published here.

The [solver architecture](/architecture/trading-systems/solver) and [Salus project overview](/work/salus) provide the deeper context.
